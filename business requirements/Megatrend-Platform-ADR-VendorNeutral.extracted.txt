How to read this document
This ADR is intentionally vendor-neutral and cost-neutral. It captures what the platform must do, 
the architectural decisions that need to be made, and the options worth weighing — but it does
not lock in specific products, vendors, hosting regions, or pricing.
The technology and cost columns are deliberately left blank, marked [TBD]. They are to be filled in
by Andrew together with an AI/engineering lead during a shaping phase, once the principles in this
ADR are agreed.
Where this document says "the platform" or "the system", it refers to a behaviour the solution must
deliver, not a specific tool.

1. Context
Megatrend operates as a commercial property and lease management partner. Today's workflow is
heavily manual — documents arrive over email, cataloguing happens in spreadsheets, and renewal
sourcing relies on account-manager memory.
The proposed platform is a dual-sided multi-tenant SaaS application: a Client Portal for tenant-
side users and a Megatrend CRM Portal for account managers, backed by an AI analysis engine
and an automated property-matching capability.

2. Goals & Non-Goals
2.1 Goals
Eliminate the email-as-CRM workflow for client document intake.
Compress the time between a document arriving and an actionable insight from days to minutes.
Catch every approaching lease expiry early enough to source alternatives proactively.
Build a structured longitudinal record of every client portfolio (a long-term data moat).
Give Megatrend account managers a credible workspace to handle more clients without
sacrificing quality.
2.2 Non-Goals (for this ADR / MVP)
Detailed UI/UX design — handled in a separate design brief.
Pricing and packaging of the service to clients.
Customer-success operating model.
Replacing Megatrend's existing accounting or invoicing systems.

3. Architectural Principles
The principles below are technology-independent and should guide all subsequent decisions. They
are the things we do not want to compromise on, even if vendor specifics change.
# Principle Implication
P1 Multi-tenant by
construction
Tenant isolation is enforced at the data layer, not in application code
alone. No design that requires the application to "remember" to filter by
tenant is acceptable.
P2 POPIA-aligned data
residency
Personal information stored in-region (South Africa) by default. Any cross-
border processing must be specifically justified and contractually covered.
P3 Human-in-the-loop AIThe AI never publishes to a client without account-manager review. AI is a
force multiplier, not an autonomous publisher.
P4 Async pipelines,
responsive UIs
Heavy work (OCR, AI, scraping) is background. User-facing surfaces stay
snappy and receive updates via real-time events.
P5 Tamper-evident audit
trail
Every meaningful action is logged immutably. Records reference the hash
of the previous record so any modification is detectable.
P6 Pluggable
infrastructure
No vendor lock-in at the architecture level — provider-specific calls are
abstracted behind internal interfaces so a vendor can be swapped without
rewriting business logic.
P7 Compliance-first
scraping
Partner APIs preferred. Any scraping is rate-limited, robots.txt-respecting,
identified in User-Agent, and has a per-source kill switch. Legal review
precedes each new source going live.
P8 Observability from day
one
Structured logs, metrics, and traces in the MVP — not a Phase 2 retrofit.
P9 Idempotency & retriesEvery pipeline step is safe to re-run. Failures retry; persistent failures land
in a dead-letter queue and alert the on-call engineer.
P10 Sensible defaults,
configurable per tenant
Lease-expiry horizons, notification preferences, and similar parameters
can be tuned per client without code changes.
4. System Components (capability view)
This is a capability-level description — what each part must do — without naming products.
4.1 Client Portal
Multi-tenant web experience accessible to client users.
Authenticated access scoped to their own tenant only.
Document upload (single and bulk; desktop and mobile-camera capture).
Dashboard with portfolio summary, leases, key dates, and curated insights.
Notification of meaningful events (new insight published, renewal options ready).

Roles within the client organisation (e.g. admin, finance, property manager).
Form-factor decision (D1): Web app · Progressive Web App · Native mobile · Combination —

4.2 Megatrend CRM Portal
Internal workspace for account managers and Megatrend admins.
Inbox of new client uploads and AI-drafted insights, ranked by urgency.
Property and lease record management.
Searchable document repository (full-text and semantic).
Curation interface to edit, approve, reject, and publish insights to a client dashboard.
Tenant administration (create client, assign team, configure thresholds).
Audit log explorer for any tenant the user is authorised to view.
4.3 Document Ingestion & OCR Capability
Accept any document the client uploads (PDF, image, scanned).
Validate (file type, size, malware).
Extract structured data (key-value pairs, tables) from invoices, utility statements, lease
agreements.
Surface low-confidence extractions for human review rather than silent failures.
OCR strategy decision (D2): managed cloud OCR · self-hosted OCR · LLM vision ·
combination — see §5.
4.4 AI Analysis Engine
Classify incoming documents (invoice, utility, lease, statement, other).
Extract structured data; normalise vendor names, units, tariff codes.
Detect anomalies in spending patterns and utility usage.
Forecast utility costs and other recurring spend.
Generate plain-language narrative insights with linked source documents and a self-reported
confidence score.
Improve over time from account-manager edits and rejections.
Model strategy decision (D3): which LLM(s), which forecasting approach, which embedding
model — see §5.
4.5 Property Scout
Watch every lease across every tenant; alert when within a configurable renewal horizon.
Source comparable properties via official partner APIs where available.

Fall back to compliant scraping of public listings only when APIs are insufficient.

4.6 Workflow Orchestration
Coordinate the document upload pipeline (validate → classify → extract → index → notify →

4.7 Real-time Event Bus
Tenant-scoped channels for live updates between client and CRM portals.
JWT-secured; each tenant can only subscribe to its own events.
Survives reconnection; degrades gracefully to polling if the connection is lost.
Real-time delivery decision (D6): managed service · self-hosted · other — see §5.
4.8 Identity & Tenancy
Multi-tenant identity with organisations, roles, and (for enterprise clients) SSO support.
MFA mandatory for Megatrend staff; optional for client users.
Tenant ID resolved from the authenticated session and propagated to every downstream call.
Identity provider decision (D7): managed identity service · self-hosted · build · combination —
see §5.
4.9 Data Layer
Primary database for structured records (tenants, properties, leases, documents metadata,
insights, audit).
Object storage for original document files.
Index for full-text and semantic search.
Cache and queueing for the async pipelines.

4.10 Audit & Compliance
Tamper-evident log of every meaningful action (client, Megatrend, system, admin, infrastructure).
Mirror to a write-once archive with long retention.
Queryable from the CRM for day-to-day inspection.
Exportable per tenant for client transparency and POPIA requests.

Structured logs, metrics, and traces from every component.
Alerting on error rates, latency anomalies, and failed jobs.
Runbooks for the on-call engineer.

In-region hosting (South Africa) for tenant data.
Infrastructure-as-code so environments can be recreated and audited.
Separate environments for development, staging, production, and audit logs.
Hosting decision (D10): which cloud / region / infrastructure model — see §5.

Document files (original uploads) + extracted structured data
AI insights (draft, curated, published)
Audit events (all of the above + system / admin activity)
8.2 POPIA controls
Control Description
Lawful basis Captured in client agreement at onboarding.
Data residency South Africa primary; any cross-border processing covered by processor agreements.
Encryption At rest (customer-managed keys) and in transit (TLS).
Access RBAC + tenant isolation enforced at the data layer.
Retention Defined per data class; documented in a Retention Schedule.
Data-subject rightsExport and deletion supported within 30 days.
Breach responseDocumented playbook; regulator notification process.
Sub-processors Each vendor signed under a processor agreement; list maintained and disclosed.
